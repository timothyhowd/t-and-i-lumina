# Lumina PoC — Limitations & Guardrails

> ⚠️ **THIS IS A PROTOTYPE.** Lumina is in proof-of-concept phase. Nothing it produces is legally vetted. No clause is approved by counsel. No regulation citation is verified. Documents generated by this system are **not for execution**.

This document is the single source of truth for what the PoC does and does not guarantee. It is linked from the app banner and from every generated document. If you see Lumina output without a `Prototype — Not for Production Use` banner, something is wrong; do not trust the output.

---

## What the PoC is

- A working demonstration of layered HR document generation across multiple countries and brands (DoorDash / Wolt / Deliveroo).
- A vehicle for a side-by-side accuracy comparison with the consulting firm's process.
- A scaffold for the eventual production system — the architecture is real; the data is partial.

## What the PoC is **not**

- A legally-vetted document generator.
- A replacement for specialist review.
- Connected to real HRIS systems (Greenhouse, Workday, DocuSign). All integrations are mocked with realistic shims.
- Approved by legal, compliance, security, or privacy. None of the above have signed off on outputs.

---

## Guardrails

### Data layer (what lives on disk)

- **No legal citations in committed artifacts.** Slot schemas carry intent notes ("statutory requirement: notification of key terms") but never section references like `§4` until counsel has verified each one. We chose option (ii) — strip citations entirely until legal provides them — over option (i) "save with `⚠ unverified` markers." The risk of someone forgetting the markers and trusting a placeholder citation is too high.
- **Templates in `corpus/templates/` are slot schemas only**, not legal prose. Real prose belongs to legal; we describe its shape and intent.
- **Raw `.docx` extracts live in `corpus/raw/` and are `.gitignore`d.** They contain real legal text and should never be committed.
- **Mock employees use obviously-fake personas**: `Test Tester`, `Sample Employee A`, `Demo Persona 1`. No real names, not even hypothetical ones a stranger could mistake for a real person.
- **Real entity addresses are fine** (they are public). Mock employees never live at real addresses.

### Output layer (every generated doc)

- **Watermark on every page**: diagonal, ~40% opacity, reading `LUMINA PROTOTYPE — NOT FOR EXECUTION — [timestamp]`. Hard-coded; cannot be toggled off.
- **Output filename convention**: every generated file starts with `POC-DRAFT-`.
- **Provenance block** (template version + corpus snapshot + regulation snapshot) is required on every doc and printed at the end. If any component is missing, generation is **blocked**, not warned.
- **Citation rendering when unverified**: any clause whose citation is not yet legal-approved renders as `⚠ Citation pending legal review — clause not for execution`. This way the doc is visibly incomplete; nobody can mistake it for finished work.

### Surface layer (the app)

- **Persistent red banner** on every screen: `Prototype — Not for Production Use`.
- **Env flag tripwire**: the app refuses to generate any document unless `LUMINA_POC_MODE=true` is set in the environment. Silly but real — anyone forking and trying to deploy must opt in deliberately.
- **No "export without watermark" path.** Ever. Not even for a clean demo screenshot.
- This file is linked from the banner so every viewer can read it.

### Failure principle

- **Fail closed.** Any uncertainty — missing slot, unverified citation, unknown legal entity, missing template version — refuses to generate and surfaces the gap. Never silently guess.

### Process layer (one bullet)

- Every demo viewer is told once, up front: *"This is a prototype. Nothing it produces is legally vetted."* Same line, every audience, every time, regardless of seniority. The banner reinforces what was said out loud.

## Known PoC gaps

These are gaps the PoC is built around, not bugs to fix:

- **Country coverage:** Only Finland has populated templates in the shared Drive (the Lumina folder). The architecture is multi-country-ready and accepts USA / UK / Brazil / Germany etc. — but the *content* is FIN-only until other countries' templates are sourced.
- **Consulting-firm side-by-side compare:** the `Z - EY` folder is empty. We have no consulting-firm outputs to benchmark against yet.
- **Regulation index:** there is no live Finlex / GOV.UK / state-API integration in the PoC. The Template Management agent does not yet scan for regulation changes.
- **Real HRIS:** Greenhouse / Workday / DocuSign are mocked. The mocks are shaped like the real API responses so the data-collection agent's swap to production is mechanical.
- **Legal review:** zero clauses have been reviewed by counsel. Every clause intent in the schema is a description of what a clause is *for*, not what it *says*.

## What we will do *before* anything generated by this PoC could be considered for real use

A non-exhaustive checklist:

1. Engage country-specific employment counsel for each in-scope jurisdiction.
2. Have counsel verify every clause intent against the real statute and produce approved clause text.
3. Have counsel approve a versioned template; record approver, date, and scope.
4. Add a regulation-watch process (the Template Management agent's job) and verify it does not surface false positives often enough to be ignored.
5. Conduct a security review covering API key handling, PII storage, and integration with Workday / DocuSign.
6. Conduct a privacy review covering personal data minimization and retention.
7. Pilot with a small number of real specialists on non-execution drafts before any live use.

Until all of the above happen, the watermark and banner stay on. No exceptions.
